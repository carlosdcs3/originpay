## Correções implementadas

### 1. Proteção do DLQ reprocess em API

Arquivo: `routes/api.php`

- `POST /api/admin/webhooks/dead-letters/{id}/reprocess` passou a exigir `admin.permission:webhooks.dlq.reprocess`.
- O middleware valida guard `admin` e permissão Spatie granular.
- Merchant e usuário comum recebem `403`.
- Admin autorizado recebe `200` quando o DLQ existe e o motivo é informado.

### 2. Middleware administrativo granular

Arquivo criado: `app/Http/Middleware/EnsureAdminPermission.php`

Responsabilidades:

- Falhar fechado quando não há admin autenticado no guard `admin`.
- Falhar fechado quando o admin não possui a permissão exigida.
- Retornar `403` para tentativas de usuário comum/merchant.

### 3. RBAC granular em rotas administrativas sensíveis

Arquivo: `routes/admin.php`

Permissões adicionadas:

| Ação | Permissão |
| --- | --- |
| Settlement pay | `settlements.pay` |
| Ajuste manual de saldo | `balances.adjust` |
| Gateway credentials | `gateway-credentials.manage` |
| Alteração de taxas | `gateway-fees.manage` |
| Alteração de routing/limites | `gateway-limits.manage` |
| API credentials generate | `api-credentials.issue` |
| API credentials rotate | `api-credentials.rotate` |
| API credentials revoke | `api-credentials.revoke` |
| Webhook admin mutation | `webhooks.manage` |
| DLQ reprocess | `webhooks.dlq.reprocess` |

Essas ações não dependem mais apenas de `isAdmin`/middleware genérico.

### 4. Seeder RBAC atualizado

Permissões adicionadas:

- `Operações`: reprocessamento DLQ.
- `Developer`: credenciais, webhooks e DLQ técnico.

### 5. API Key scopes reais

Arquivo criado: `app/Http/Middleware/EnsureApiKeyScope.php`

Arquivo alterado: `bootstrap/app.php`

Alias registrado:

- `api.scope`

Arquivo alterado: `routes/api.php`

Scopes aplicados:

| Rota | Scope |
| --- | --- |
| `POST /api/v1/charges` | `charges.write` |
| `GET /api/v1/charges` | `charges.read` |
| `GET /api/v1/charges/{id}` | `charges.read` |
| `POST /api/v1/payments` | `charges.write` |
| `GET /api/v1/payments` | `charges.read` |
| `GET /api/v1/payments/{id}` | `charges.read` |
| `POST /api/v1/refunds` | `refunds.write` |
| `POST /api/v1/payouts` | `settlements.write` |
| `GET /api/v1/payouts` | `settlements.read` |
| `GET /api/v1/balance` | `settlements.read` |
| `POST /api/v1/customers` | `customers.write` |
| `GET /api/v1/customers/{id}` | `customers.read` |
| `POST /api/v1/webhooks/test` | `webhooks.write` |

Compatibilidade:

- `permissions = ['*']` ainda é aceito como fallback compatível para chaves legadas existentes.
- Novos testes cobrem negação por scope ausente e permissão por scope presente.

### 6. Auditoria obrigatória

Arquivo: `app/Http/Controllers/Webhook/GatewayWebhookController.php`

Ação `reprocess()` agora exige `reason` e registra audit log com:

- usuário admin (`id`, `name`, `email`)
- timestamp
- motivo
- recurso
- alteração before/after
- IP
- correlation id

Arquivo: `config/logging.php`

Canais adicionados:

- `audit`
- `gateway`

O canal `audit` elimina falhas do `AdminAuditMiddleware` e permite trilha administrativa centralizada em `storage/logs/audit.log`.

### 7. Regressão de webhook/DLQ corrigida

Arquivo: `tests/Feature/GatewayWebhookValidationTest.php`

Problema encontrado na regressão completa:

- O teste `test_valid_webhook_is_accepted_and_queued` falhava com `500` por ausência da tabela `webhook_events` no contrato local do teste.

Correção:

- O teste agora cria uma tabela `webhook_events` mínima compatível quando ela não existe, preservando o contrato de validação de webhook e sem alterar fluxo de produção.

## Vulnerabilidades eliminadas

- Rota administrativa de DLQ reprocess exposta em API.
- Ausência de permissão granular para DLQ reprocess.
- Ausência de escopos reais para API keys nas principais rotas públicas v1.
- Dependência de middleware administrativo genérico para settlement, saldo, gateway credentials, taxas, limites, API credentials e webhook admin.
- Ausência de canal `audit` usado pelo middleware administrativo.
- Auditoria incompleta no DLQ reprocess.

## Riscos restantes

1. Pint global ainda falha por baseline histórico amplo fora dos arquivos alterados.
2. Scopes foram aplicados às rotas públicas v1 principais; rotas adicionais de `customer-subscriptions` ainda devem receber matriz granular em fase posterior.
3. Auditoria sensível foi reforçada no DLQ reprocess; recomenda-se evoluir todas as ações sensíveis para trilha persistida/imutável em banco, além de log em arquivo.
4. `permissions = ['*']` foi mantido por compatibilidade; deve haver plano de migração para remover wildcard em produção madura.
5. Dual control/step-up ainda deve ser aprofundado para settlement pay, ajuste de saldo e credenciais críticas.
