No workspace atual, `docs/production/` não existia antes desta fase; foi criado com `09-gateway-architecture.md` e `00-architecture.md`. Em `docs/audit/`, o arquivo disponível era `R2-webhooks-dlq-idempotency.md`.

## Source of truth oficial

A camada oficial da OriginPay para gateways é **`app/Gateway`**.

`app/Payment` e `app/Payment/Modern` são classificados como legado/migração. Eles podem permanecer temporariamente enquanto houver referências runtime, mas não podem receber novos PSPs ou métodos de pagamento.

### Justificativa baseada em evidências do código

- `app/Gateway/GatewayResolver.php` já contém resolução por operação, fallback, KYC, maintenance, circuit breaker/health score.
- `app/Gateway/GatewayManager.php` já resolve adapters legados e providers novos.
- `app/Gateway/Contracts/GatewayProviderInterface.php` define contrato oficial de provider (`getIdentifier`, `sendRequest`, `checkHealth`).
- `app/Gateway/Contracts/GatewayWebhookValidatorInterface.php` define contrato de validação/normalização de webhook.
- `app/Gateway/Contracts/Data/*` contém DTOs de fronteira (`GatewayResponse`, `GatewayWebhookData`, `GatewayCredentials`, etc.).
- `app/Gateway/Http/*` contém pipeline HTTP, middlewares, transporte e response mapper.
- `app/Gateway/Security/*` contém autenticação extensível por driver.
- `app/Gateway/Metrics`, `Events`, `CircuitBreaker`, `Policies` compõem a infraestrutura de resiliência/observabilidade.

## Diagnóstico resumido

Há três camadas coexistentes:

1. `app/Payment`: muitos providers antigos por diretório; modelo legado baseado em `PaymentGatewayInterface`/`PaymentGateway`.
2. `app/Payment/Modern`: tentativa intermediária com DTOs próprios, factory própria e provider EFI moderno.
3. `app/Gateway`: camada mais completa, com contratos, providers, webhooks, pipeline HTTP, segurança, métricas, circuit breaker e resolver.

Além disso, há uma quarta zona de duplicidade fora do escopo nominal, mas relevante:

- `app/Services/Gateways/*`, com adapters/manager/registry paralelos.

## Arquitetura alvo

Ver especificação obrigatória em `docs/production/09-gateway-architecture.md`.

Resumo:

```text
Application Service
 -> GatewayResolver
 -> GatewayManager
 -> app/Gateway/Providers/{Provider}
 -> GatewayHttpClient + pipeline
 -> PSP
 -> GatewayResponse
 -> domínio financeiro
 -> ledger/audit/observability
```

## Fluxos oficiais

### Fluxo financeiro

1. Serviço de aplicação recebe intenção financeira.
2. Resolve operação via `GatewayResolver`.
3. Executa provider via `GatewayManager`.
4. Provider retorna DTO canônico.
5. Serviço financeiro decide estado e ledger em transação.
6. Observabilidade registra resultado.

### Fluxo autenticação PSP

1. Provider declara mecanismo necessário.
2. `GatewayAuthenticationService` usa driver registrado.
3. Driver lê credenciais seguras.
4. Header/token/certificado é aplicado pelo pipeline.

### Fluxo webhook

1. PSP chama endpoint compatível.
2. Assinatura/timestamp/shape são validados.
3. Evento é persistido antes da fila.
4. Replay é bloqueado.
5. Job processa evento idempotente.
6. Falha final vai para DLQ canônica.

### Fluxo settlement

1. Provider normaliza settlement/report.
2. Serviço de settlement aplica regra de domínio.
3. Conciliação compara esperado vs PSP.
4. Ledger recebe lançamentos idempotentes.
5. Divergências geram auditoria/alerta.

### Fluxo ledger

1. Nenhum provider escreve saldo.
2. Serviços financeiros transacionais escrevem ledger.
3. Chaves idempotentes impedem duplicidade.
4. Correlação provider/event/charge é obrigatória.

### Fluxo observabilidade

1. Correlation id no gateway pipeline.
2. Métrica por provider/operação/latência/status.
3. Log estruturado sem segredo.
4. Circuit breaker e DLQ monitorados.

## Plano de migração incremental

1. Congelar novas integrações fora de `app/Gateway`.
2. Inventariar providers ativos em banco/config/admin.
3. Priorizar PSPs ativos por volume/risco.
4. Para cada provider ativo legado:
   - criar provider canônico em `app/Gateway/Providers/{Provider}`;
   - criar webhook validator canônico;
   - mapear DTOs para `Contracts/Data`;
   - adicionar testes de paridade;
   - habilitar por feature flag/roteamento;
   - observar métricas/DLQ;
   - remover referência ao legado.
5. Migrar `PaymentService`/`ProcessWebhookJob` que ainda dependem de `app/Payment`/`app/Payment/Modern`.
6. Consolidar managers/registries duplicados.
7. Migrar EFI v1 webhook para fluxo canônico.
8. Remover classes legadas somente após critérios objetivos.

## Riscos da migração

- Quebra de PSP ainda ativo por referência indireta em DB/config.
- Divergência de payload/assinatura entre provider legado e novo.
- Webhook duplicado durante coexistência de endpoints.
- Lançamento financeiro duplicado se idempotência não for preservada.
- Perda de observabilidade se métricas antigas não forem mapeadas.
- Incompatibilidade de DTOs Modern com DTOs canônicos.

## Testes obrigatórios antes de remover código legado

Para cada PSP migrado:

1. Contrato de provider por operação suportada.
2. Paridade de payload legado vs novo para operação crítica.
3. Sucesso PSP.
4. Erro PSP mapeado.
5. Timeout/retry.
6. Circuit breaker.
7. Webhook válido.
8. Webhook inválido.
9. Replay/idempotência.
10. Falha final para DLQ.
11. Reprocessamento de DLQ preservando payload/headers/assinatura/timestamp.
12. Settlement/conciliação se aplicável.
13. Ledger: não duplicidade financeira.
14. Observabilidade: métrica/log/correlation id.
15. Rollback/feature flag quando aplicável.

## Critérios objetivos para considerar a migração concluída

1. Busca por `App\Payment\` não retorna callers runtime.
2. Busca por `Payment\Modern` não retorna callers runtime.
3. Todos os PSPs ativos em DB/config apontam para `app/Gateway`.
4. `GatewayResolver` e `GatewayManager` são o único caminho de resolução/execução.
5. Webhooks inbound financeiros usam fluxo R2 (`webhook_events` + DLQ canônica).
6. Nenhum provider escreve ledger/saldo diretamente.
7. Testes obrigatórios passam para todos os PSPs ativos.
8. Observabilidade padronizada em todos os providers.
9. Documentação `docs/production/09-gateway-architecture.md` é referenciada em PRs de gateway.
