## 4. Comparação e conflitos

```text

```

Como a regra foi **não apagar nada sem comparar**, os conteúdos vindos de `core/docs` foram preservados em área explícita de conflito:

```text

../docs/audit/core-docs-conflicts/production/00-architecture.md
```

Os arquivos canônicos já existentes foram mantidos em seus caminhos originais:

```text

```

Conflitos pendentes de resolução humana/técnica:

1. Comparar `../docs/design-system.md` com `../docs/audit/core-docs-conflicts/design-system.md`.
2. Comparar `../docs/production/00-architecture.md` com `../docs/audit/core-docs-conflicts/production/00-architecture.md`.
3. Decidir se o conteúdo preservado em `core-docs-conflicts` deve ser incorporado ao arquivo canônico ou arquivado como histórico.

## 8. Referências internas

Referências textuais a `core/docs` foram preservadas quando fazem parte do histórico/diagnóstico do R3.1, pois descrevem a causa do problema.

Regra operacional atualizada para referências futuras:

- Dentro de `core`, usar `../docs/...` para documentação.
- Em relatórios dentro da árvore canônica, caminhos relativos podem ser escritos como `../docs/...` quando o contexto for execução a partir de `core`, ou como `docs/...` quando o contexto for a raiz superior do repositório.
- Não usar `core/docs` como destino de novos relatórios.

## 9. Regra oficial para próximas fases

A partir desta fase:

1. Documentação oficial da OriginPay fica exclusivamente em:

```text
E:\projetos\DigiKash v1.0.5\DigiKash v1.0.5\docs
```

2. Quando trabalhando dentro de `core`, qualquer leitura/escrita documental deve usar:

```text
../docs/...
```

3. É proibido criar novos documentos em:

```text
E:\projetos\DigiKash v1.0.5\DigiKash v1.0.5\core\docs
```

4. Se `core/docs` reaparecer no futuro, deve ser tratado como desvio e consolidado novamente em `../docs` antes de iniciar nova fase.

5. Conflitos devem ser preservados em `../docs/audit/core-docs-conflicts/` até comparação e decisão explícita.

## 10. Conclusão

A documentação foi canonicalizada em `../docs`. Os documentos produzidos em `core/docs` durante R2.5, R3 e R3.1 foram movidos ou preservados em área de conflito quando já existia arquivo canônico com conteúdo diferente. `core/docs` ficou vazio e foi removido. Próximas fases devem usar exclusivamente `../docs` para documentação.
